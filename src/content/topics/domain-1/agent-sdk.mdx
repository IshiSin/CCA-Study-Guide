Say you want to build a Python bot that can check the weather and book calendar events. Not just answer questions about them — actually do it.

That's what the `Claude Agent SDK` (`anthropic.agents`) is for. You write normal Python functions, tell Claude they exist, and Claude will call them when needed. The SDK handles all the plumbing in between.

---

## Step 1 — Give Claude something it can do: `@tool`

A tool is just a Python function with `@tool` above it. That one line tells the SDK: "Claude is allowed to call this."

```python
from anthropic.agents import tool

@tool  # registers this function — without this, Claude can't see or call it
def get_weather(city: str) -> str:
    """Return the current weather for a city. Example: get_weather("London")"""
    # Claude reads the docstring above to decide when to call this function
    return f'{{"city": "{city}", "temp": 18, "conditions": "cloudy"}}'
```

Two things matter here:

**The `@tool` decorator** — without it, this is just a regular Python function Claude knows nothing about. Adding `@tool` registers it with the SDK.

**The docstring** (`"""..."""`) — this is what Claude actually reads to decide whether and how to call the tool. Think of it as the instruction label on a button. If it's vague, Claude presses the wrong button or none at all.

<ExamTip>
The exam tests docstring quality. A good tool docstring answers three things: when to use it, what to pass in, and what comes back. Vague descriptions cause wrong or missed tool calls — Claude's decision is only as good as the description it reads.
</ExamTip>

## Step 2 — Run it: `Agent`

`Agent` is the object that runs the loop. You hand it your tools and a goal — it keeps calling Claude and running tools until the task is done.

```python
from anthropic.agents import Agent
import anthropic

agent = Agent(
    client=anthropic.Anthropic(),
    model="claude-opus-4-5",
    system="You are a helpful scheduling assistant.",  # Claude's role and any rules
    tools=[get_weather],  # the @tool functions Claude is allowed to call
)

# starts the loop: Claude reasons, calls tools, reads results, repeats
agent.run("What's the weather like in Tokyo?")
print(agent.result())  # the final answer, once Claude has no more tool calls to make
```

What happens behind the scenes:
1. Claude reads your message and decides to call `get_weather("Tokyo")`
2. The SDK runs your function and gets the result
3. Claude reads the result and writes a final response
4. Loop ends — no more tool calls needed

## Step 3 — Watch and control the loop: `AgentHooks`

`AgentHooks` lets you plug into each stage of the loop — to log what's happening, stop it if something goes wrong, or block a tool call before it executes.

You subclass it and only override the parts you care about:

```python
from anthropic.agents import AgentHooks

class MyHooks(AgentHooks):

    # called before each iteration — use this to cap how long the agent can run
    def on_iteration_start(self, iteration: int, context):
        if iteration > 15:
            raise RuntimeError("Agent ran too long — stopping.")

    # called before every tool call — this is where you allow, block, or modify it
    def on_tool_call(self, tool_name: str, tool_input: dict) -> dict | None:
        print(f"Claude wants to call: {tool_name}")
        # raise an exception → block the call entirely
        # return a new dict   → override the inputs Claude chose
        # return None         → let it proceed as normal
        return None

    # called once when the loop ends, whatever the reason
    def on_stop(self, reason: str, iterations: int, result: str | None):
        print(f"Done after {iterations} steps.")

agent = Agent(
    client=client, model="claude-opus-4-5",
    system="You are a helpful assistant.",
    tools=[get_weather], hooks=MyHooks(),  # attach hooks here
)
```

### Guardrails belong in hooks, not in the system prompt

You might be tempted to write `"Never access internal URLs"` in the system prompt. Claude will usually follow it — but not always. A hook runs in your Python code. It cannot be overridden by anything Claude says.

```python
class SafetyHooks(AgentHooks):
    def on_tool_call(self, tool_name: str, tool_input: dict) -> dict | None:
        if tool_name == "fetch_url" and "internal.company.com" in tool_input.get("url", ""):
            raise PermissionError("Internal URLs are blocked.")  # Claude never sees this URL
        return None  # all other calls proceed normally
```

<ExamTip>
The exam tests this directly: use hooks for security rules, not system prompt instructions. Hooks are programmatic — they always fire. System prompt instructions rely on the model choosing to follow them.
</ExamTip>

## Step 4 — Delegate complex subtasks: `SubAgent`

Sometimes one step of your task is itself complex enough to need its own loop — multiple tool calls, its own context, its own iteration limit. That's what `SubAgent` is for.

```python
from anthropic.agents import Agent, SubAgent

class ResearchOrchestrator(Agent):
    async def deep_research(self, topic: str) -> str:
        sub = SubAgent(
            client=self.client,
            model="claude-haiku-4-5",  # use a cheaper model — the subtask is narrow
            system="You are a research specialist.",
            tools=[search_web, read_document],  # only the tools this subtask needs
            max_iterations=5,  # its own cap, separate from the parent agent's
        )
        result = await sub.run_async(topic)  # runs its own full loop independently
        return result.text  # the parent only sees this final string — not the intermediate steps
```

The parent agent never sees the subagent's intermediate steps — just the final result. Each subagent has its own isolated context window.

Use `SubAgent` when the subtask needs multiple tool calls. Use a plain `client.messages.create()` call when one response is enough.

---

<KeyTakeaway>
- `@tool` makes a Python function callable by Claude — the docstring is what Claude reads to decide when and how to call it
- `Agent` runs the loop: Claude reasons → tool runs → result feeds back → repeat until done
- `AgentHooks` plugs into the loop for logging, cost limits, and safety checks — `on_tool_call` can block or modify any tool call before it executes
- Guardrails in hooks always run; guardrails in the system prompt rely on model compliance
- `SubAgent` delegates a subtask that needs its own loop; use `messages.create` for single-turn subtasks
</KeyTakeaway>
