An `agentic loop` is the fundamental execution pattern of any Claude-based agent. Rather than a single prompt → response, the agent iterates: it reasons about a goal, takes an action (tool call), observes the result, then decides whether to continue or stop.

## The Loop Lifecycle

Every agentic loop passes through four phases per iteration:

### Phase 1 — Reason

Claude receives the full conversation context (system prompt + history + tool results) and decides what to do next. It may call a tool, ask a clarifying question, or produce a final response.

### Phase 2 — Act

If Claude emits a tool call, the host application executes it. The tool might be a database query, file read, API call, or a sub-agent invocation. Execution happens `outside` Claude — the model only receives the result.

### Phase 3 — Observe

The tool result is appended to the conversation as a `tool_result` message. Claude now has updated context and re-enters the Reason phase.

### Phase 4 — Stop or Continue

The loop ends when `any` stop condition is met. If none are met, the loop continues.

## Concrete Walkthrough: Two Iterations

`Goal:` "Check if order #ORD-4521 has shipped, and email the customer if it has."

Available tools: `check_order_status`, `send_email`.

---

`Iteration 1`

`Reason` — Claude sees both tools, recognises it doesn't have order status yet, and emits a tool call:

```json
{ "name": "check_order_status", "input": { "order_id": "ORD-4521" } }
```

`Act` — The host executes `check_order_status("ORD-4521")` and gets back:

```json
{
  "status": "shipped",
  "carrier": "FedEx",
  "tracking": "7489-2341-XX",
  "customer_email": "alice@example.com"
}
```

`Observe` — This result is appended to the conversation as a `tool_result` message. Claude's context now contains the carrier, tracking number, and customer email.

---

`Iteration 2`

`Reason` — Claude now has everything it needs. It decides to send the email and emits:

```json
{
  "name": "send_email",
  "input": {
    "to": "alice@example.com",
    "subject": "Your order has shipped!",
    "body": "Order #ORD-4521 is on its way via FedEx. Tracking number: 7489-2341-XX."
  }
}
```

`Act` — Host executes `send_email(...)`.

`Observe` — Result appended: `{ "status": "sent", "message_id": "msg_8b3f" }`.

`Stop` — No more tool calls are needed. Claude emits its final text response:

> *"Done — order #ORD-4521 has shipped via FedEx (tracking: 7489-2341-XX) and I've sent a notification to alice@example.com."*

The host sees no `tool_use` block in this response, so the loop exits cleanly after 2 iterations.

<ExamTip>
The exam tests whether you know that `max_iterations must be set explicitly by the host` — it is not a parameter you pass to the Claude API. Claude itself has no built-in iteration limit. A loop without a host-enforced max_iterations can run indefinitely and exhaust your token budget.

## SDK Implementation

The `Agent` class, `@tool` decorator, and `AgentHooks` are covered in the [Agent SDK topic](/study/domain-1/agent-sdk). This section focuses on what's new: enforcing the stop conditions from the table above inside a running loop.

```python
consecutive_errors = 0  # tracks back-to-back tool failures

for event in agent.run(user_goal):

    if event.type == "iteration_start":
        # on_iteration_start in AgentHooks handles the max_iterations guard —
        # see the Agent SDK topic. Same idea, just inline here for clarity.
        pass

    elif event.type == "tool_error":
        consecutive_errors += 1
        if consecutive_errors >= 3:      # give up after 3 failures in a row
            agent.stop(reason="error_threshold_exceeded")
            break

    elif event.type == "tool_result":
        consecutive_errors = 0           # a successful call resets the streak

    elif event.type == "usage_update":
        if event.usage.input_tokens > 150_000:  # stop before hitting the context ceiling
            agent.stop(reason="token_budget_exceeded")
            break

return agent.result()
```

## Token Budget Planning

Each iteration adds to the context. Plan for it:

```python
# Check whether your loop will fit inside Claude's 200K window before you run it.
def estimate_loop_cost(system_tokens, avg_tool_tokens, avg_response_tokens, max_iterations):
    # every iteration permanently adds one tool result + one response to the context
    growth_per_iter = avg_tool_tokens + avg_response_tokens
    max_context = system_tokens + (growth_per_iter * max_iterations)
    return {
        "max_context_tokens": max_context,
        "fits_in_200k": max_context < 200_000,
    }
```

<ExamTip>
Context grows `cumulatively` in agentic loops — every tool result and response stays in the window. A loop with 20 iterations and 500-token tool results adds 10,000+ tokens. The exam tests whether you account for this in system design questions.
</ExamTip>

<KeyTakeaway>
- The agentic loop is: Reason → Act → Observe → repeat until stop condition
- `Stop conditions must be enforced by the host` — Claude has no built-in iteration limit
- Always implement: max_iterations, error threshold, and token budget guards
- Lifecycle hooks are the correct place for logging, rate limiting, and audit trails — not tool implementations
- Context grows cumulatively each iteration — budget for this in system design
</KeyTakeaway>
