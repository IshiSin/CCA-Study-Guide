`Task decomposition` is the process of breaking a complex goal into subtasks that can be efficiently executed by agents. Poor decomposition leads to wasted context, coordination overhead, and hard-to-debug failures.

## Three Decomposition Patterns

### Sequential (Pipeline)

Tasks must execute in order — each step's output is the next step's input.

`Use when:` Tasks have strict data dependencies (B needs A's output)

```python
def sequential_pipeline(raw_document: str) -> dict:
    """Extract → Validate → Enrich: each step feeds the next."""

    # Step 1: Extract structured data
    extraction = run_subagent(
        "Extract all invoice fields as JSON",
        {"document": raw_document}
    )

    # Step 2: Validate the extraction (needs step 1's output)
    validation = run_subagent(
        "Validate the extracted invoice data against business rules",
        {"extracted_data": extraction}
    )

    # Step 3: Enrich (needs both prior steps)
    enriched = run_subagent(
        "Enrich validated invoice with vendor metadata",
        {"validated_data": validation, "raw": raw_document}
    )

    return {"result": enriched, "stages": 3}
```

### Parallel (Fan-out / Fan-in)

Independent tasks run simultaneously; results are aggregated.

`Use when:` Tasks are independent and latency matters

```python
import asyncio

async def parallel_research(topics: list[str]) -> str:
    """Research multiple topics simultaneously, then synthesize."""

    # Fan-out: all research tasks run in parallel
    research_tasks = [
        asyncio.create_task(
            run_subagent_async("Research this topic thoroughly", {"topic": t})
        )
        for t in topics
    ]
    results = await asyncio.gather(*research_tasks)

    # Fan-in: aggregator synthesizes all results
    combined = "\n\n".join(
        f"### {topic}\n{result}"
        for topic, result in zip(topics, results)
    )
    return run_subagent(
        "Synthesize these research sections into a coherent report",
        {"sections": combined}
    )
```

### Hierarchical (Tree)

Sub-coordinators manage clusters of related tasks, reporting up to a root coordinator.

`Use when:` Task count is large (10+) or task groups have distinct expertise requirements

```python
def hierarchical_agent(enterprise_goal: str) -> str:
    # Root coordinator creates sub-coordinators per domain
    domains = decompose_into_domains(enterprise_goal)

    domain_reports = {}
    for domain in domains:
        # Each sub-coordinator manages its own team of leaf agents
        domain_reports[domain["name"]] = domain_coordinator(
            goal=domain["description"],
            tools=domain["allowed_tools"],
        )

    # Root aggregates domain reports
    return synthesize_final_report(enterprise_goal, domain_reports)
```

## Context Budget Per Subtask

Each subtask gets its own context window, but you must plan the budget.

```python
# Estimated token costs per subtask type (input = what Claude reads, output = what it writes)
SUBTASK_BUDGETS = {
    "extraction":     {"input": 8_000,  "output": 2_000},
    "analysis":       {"input": 16_000, "output": 4_000},
    "generation":     {"input": 4_000,  "output": 8_000},
    "review":         {"input": 20_000, "output": 2_000},
    "classification": {"input": 2_000,  "output": 500},
}

# Given a list of subtasks, calculate total cost and whether to run them in parallel or sequence.
def plan_decomposition(subtasks: list[dict], limit: int = 200_000) -> dict:
    cost = lambda t: SUBTASK_BUDGETS.get(t["type"], {}).get("input", 10_000)
    total = sum(cost(t) for t in subtasks)      # sequential: all costs added together
    peak  = max(cost(t) for t in subtasks)      # parallel: only the heaviest task matters
    return {
        "fits_in_window": peak < limit,
        "recommendation": "parallel" if total > limit * 0.5 else "sequential",
    }
```

`Too fine:` Coordination overhead dominates, excessive API calls
`Right size:` Each subtask can be completed reliably with the tools and context it receives

<KeyTakeaway>
- Three patterns: Sequential (ordered dependencies), Parallel (independent), Hierarchical (large/complex task trees)
- Decompose when: tasks are independent, context would overflow, different model tiers would help, or independent retry is valuable
- Keep monolithic when: subtasks are heavily interdependent, task is simple, or coordination overhead exceeds benefit
- Budget context per subtask: plan for input + output tokens for each agent invocation
- Right-size subtasks: single clear output, ≤ 3 tools, completable in ≤ 5 iterations
</KeyTakeaway>
