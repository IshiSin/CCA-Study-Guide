Even with well-designed schemas, Claude can occasionally produce invalid output — malformed JSON, missing required fields, or values outside allowed ranges. `Validation and retry` patterns make output pipelines robust without requiring perfect output on every call.

## The Parse-Then-Validate Pattern

Validation has two stages:

1. `Parse` — can the text be deserialized? (`json.loads()` succeeds)
2. `Validate` — does the parsed object match the expected schema?

Both can fail independently. A string `"true"` parses fine but may fail if a boolean was required:

```python
import json, anthropic
from pydantic import BaseModel, ValidationError
from typing import Literal

client = anthropic.Anthropic()

class SentimentResult(BaseModel):
    sentiment: Literal["positive", "negative", "neutral"]
    confidence: float
    requires_escalation: bool

def classify_message(message: str) -> SentimentResult:
    response = client.messages.create(
        model="claude-opus-4-6", max_tokens=256,
        system="Classify the message. Return valid JSON only matching the specified schema.",
        messages=[{"role": "user", "content": message}]
    )
    raw = response.content[0].text

    # Stage 1: parse — can the text be deserialized?
    try:
        data = json.loads(raw)
    except json.JSONDecodeError as e:
        raise ValueError(f"Invalid JSON: {e}\n\nRaw: {raw}")

    # Stage 2: validate — does it match the expected schema?
    try:
        return SentimentResult(**data)
    except ValidationError as e:
        raise ValueError(f"Schema mismatch: {e}\n\nData: {data}")
```

Using Pydantic for validation catches type mismatches, missing fields, and constraint violations with structured error messages that can be fed back to Claude.

## Feeding Errors Back to Claude

The most effective retry strategy is to include the error in the next request. Claude can self-correct when it understands what went wrong:

```python
def extract_with_retry(document: str, schema_description: str, max_retries: int = 3) -> dict:
    messages = [{"role": "user", "content": f"Extract data from:\n\n{document}"}]

    for attempt in range(max_retries):
        response = client.messages.create(
            model="claude-opus-4-6", max_tokens=1024,
            system=f"Extract structured data. Return valid JSON only.\nSchema: {schema_description}",
            messages=messages
        )
        raw = response.content[0].text
        messages.append({"role": "assistant", "content": raw})  # keep Claude's output in context

        try:
            return json.loads(raw)   # success

        except json.JSONDecodeError as e:
            # Tell Claude exactly what was wrong before retrying
            messages.append({
                "role": "user",
                "content": f"JSON parse error: {e}\n\nYour output:\n{raw}\n\nReturn valid JSON only."
            })

    raise RuntimeError(f"Failed after {max_retries} attempts")
```

<ExamTip>
The exam tests the feedback loop pattern specifically. The key insight is that you append Claude's failing output and the error message to the conversation before the next attempt. This gives Claude full context about what it produced and what was wrong — far more effective than just retrying the original prompt.
</ExamTip>

## Schema Validation Retry

For schema errors (parsed but invalid), include both the error and the schema in the feedback:

```python
from pydantic import BaseModel, ValidationError

class InvoiceData(BaseModel):
    vendor: str
    invoice_number: str
    amount: float
    currency: str = "USD"
    due_date: str   # YYYY-MM-DD

def extract_invoice_with_retry(text: str, max_retries: int = 3) -> InvoiceData:
    messages = [{"role": "user", "content": f"Extract invoice data:\n\n{text}"}]

    for attempt in range(max_retries):
        response = client.messages.create(
            model="claude-opus-4-6", max_tokens=512,
            system='Return JSON only: {"vendor": str, "invoice_number": str, "amount": number, "currency": str, "due_date": "YYYY-MM-DD"}',
            messages=messages
        )
        raw = response.content[0].text
        messages.append({"role": "assistant", "content": raw})

        try:
            return InvoiceData(**json.loads(raw))   # parse + validate in one step

        except (json.JSONDecodeError, ValidationError) as e:
            # Feed the specific error back so Claude knows what to fix
            messages.append({
                "role": "user",
                "content": f"Validation failed: {e}\n\nYour output:\n{raw}\n\nCorrect and return JSON only."
            })

    raise RuntimeError(f"Could not extract valid invoice after {max_retries} attempts")
```

## Exponential Backoff for Rate Limits

Retry for validation failures is different from retry for API errors. For `429 Too Many Requests` and `529 Overloaded` errors, use exponential backoff with jitter:

```python
import time, random, anthropic

client = anthropic.Anthropic()

def call_with_backoff(messages: list, max_retries: int = 5, base_delay: float = 1.0) -> anthropic.types.Message:
    for attempt in range(max_retries):
        try:
            return client.messages.create(
                model="claude-opus-4-6", max_tokens=1024, messages=messages
            )
        except anthropic.RateLimitError:
            if attempt == max_retries - 1:
                raise
            # Exponential backoff with jitter — avoids thundering herd
            delay = min(base_delay * (2 ** attempt) + random.uniform(0, 1), 60.0)
            time.sleep(delay)
        except anthropic.APIStatusError as e:
            if e.status_code == 529:   # Overloaded — wait and retry
                if attempt == max_retries - 1: raise
                time.sleep(min(base_delay * (2 ** attempt), 60.0))
            else:
                raise   # 4xx errors are not retryable
```

<Callout variant="info" title="Validation retry vs. API error retry are different patterns">
Validation retries extend the conversation (append error messages). API error retries (rate limits, overload) repeat the same request after a delay. Never conflate these — extending the conversation for a rate limit error wastes tokens; simply waiting is correct.
</Callout>

## Max Retry Limits and Failure Modes

Always set a hard ceiling on retries. Without it, a bad schema or an unusual input can trigger infinite loops:

```python
MAX_VALIDATION_RETRIES = 3   # parse/schema failures: extend the conversation
MAX_API_RETRIES        = 5   # rate limits / overload: wait and repeat same request

# After exhausting retries, raise — don't return None or {}
# Callers need to know extraction failed, not silently get empty data
raise ExtractionError(
    f"Could not extract valid output after {MAX_VALIDATION_RETRIES} attempts",
    last_raw_output=raw,
    last_error=error_msg
)
```

### What to do on permanent failure

```python
# Option 1: raise — let the caller decide how to handle
raise ExtractionError(...)

# Option 2: return a sentinel — best for batch pipelines (don't abort on one failure)
return ExtractionResult(success=False, error=error_msg, raw_output=raw, data=None)

# Option 3: queue for human review — best when data quality is critical
send_to_human_review_queue(document=document, error=error_msg, raw_output=raw)
```

For batch pipelines processing thousands of documents, Option 2 or 3 is usually correct — you want the pipeline to continue and collect failures for later analysis, not abort on the first bad document.

## Retry with tool_use

When using tool_use, validation failures are rarer (schema is enforced by the API), but you can still get missing fields or wrong types. The same feedback pattern applies — return the tool result with an error to trigger re-extraction:

```python
def extract_with_tool_retry(document: str, max_retries: int = 3) -> dict:
    messages = [{"role": "user", "content": document}]

    for attempt in range(max_retries):
        response = client.messages.create(
            model="claude-opus-4-6", max_tokens=1024,
            tools=extraction_tools, tool_choice={"type": "any"},
            messages=messages
        )

        tool_use = next((b for b in response.content if b.type == "tool_use"), None)

        if not tool_use:
            # Claude didn't call the tool — tell it to
            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": [{
                "type": "tool_result", "tool_use_id": "none",
                "content": "No tool was called. Please call the extraction tool."
            }]})
            continue

        try:
            return validate_extraction(tool_use.input)   # validate the extracted dict

        except ValueError as e:
            # Feed back as a tool_result with is_error=True — protocol-correct way to signal failure
            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": [{
                "type": "tool_result",
                "tool_use_id": tool_use.id,
                "content": f"Validation failed: {e}. Re-extract with corrections.",
                "is_error": True
            }]})

    raise RuntimeError(f"Extraction failed after {max_retries} attempts")
```

<KeyTakeaway>
- Validation has two stages: parse (JSON deserialization) then validate (schema conformance)
- Use Pydantic or similar libraries for schema validation — they produce structured, field-level error messages
- Feed errors back into the conversation before retrying — append the error and raw output as a new user turn
- For API errors (rate limits, overload), use exponential backoff with jitter — do NOT extend the conversation
- Always set a hard max retry ceiling (3 for validation, 5 for API errors)
- For batch pipelines, return a failure sentinel rather than aborting on first error
- For tool_use retry, feed back errors as `tool_result` with `is_error: true` — this is the protocol-correct approach
</KeyTakeaway>
