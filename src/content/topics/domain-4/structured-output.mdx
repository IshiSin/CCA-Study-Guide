`Structured output` means getting Claude to return data in a predictable, machine-parseable format — typically JSON — instead of prose. The exam tests two primary mechanisms: `tool_use` schemas and direct JSON prompting.

## The Core Problem

Claude is a language model, not a JSON serializer. Without guidance, it may:
- Add explanatory text before or after JSON
- Produce different key names across calls
- Omit optional fields entirely
- Use inconsistent types (string vs. number)

The solution is to use the API's structured mechanisms rather than relying on prompt instructions alone.

---

## Method 1: tool_use for Structured Extraction

The most reliable way to get structured output is to define a tool with a JSON Schema and have Claude "call" it. This forces the output to conform to the schema:

```python
import anthropic, json
client = anthropic.Anthropic()

tools = [{
    "name": "extract_contract_metadata",
    "description": "Extract structured metadata from a contract. Call once with all fields.",
    "input_schema": {
        "type": "object",
        "properties": {
            "party_a":        {"type": "string", "description": "First contracting party"},
            "party_b":        {"type": "string", "description": "Second contracting party"},
            "effective_date": {"type": "string", "description": "Start date, ISO 8601 (YYYY-MM-DD)"},
            "duration_months":{"type": "integer","description": "Contract length in months"},
            "total_value_usd":{"type": "number", "description": "Total value in USD"},
            "contract_type":  {
                "type": "string",
                "enum": ["service", "license", "employment", "nda", "other"],  # prevents invented values
            },
            "auto_renews":    {"type": "boolean","description": "Whether the contract auto-renews"},
        },
        "required": ["party_a", "party_b", "effective_date", "contract_type"]
    }
}]

response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=1024,
    tools=tools,
    tool_choice={"type": "any"},   # force Claude to call the tool, not respond in prose
    messages=[{"role": "user", "content": f"Extract metadata:\n\n{contract_text}"}]
)

tool_use_block = next(b for b in response.content if b.type == "tool_use")
extracted = tool_use_block.input   # already a Python dict — no json.loads() needed
```

<ExamTip>
`tool_choice: {"type": "any"}` forces Claude to call one of the provided tools. Without it, Claude may choose to respond in prose instead of calling the tool. For extraction tasks where you always need structured output, always set `tool_choice`.
</ExamTip>

## tool_use Schema Design

The quality of your schema determines the quality of Claude's output. Key schema design principles:

### Use `description` fields extensively

Claude reads descriptions to understand how to fill each field. Descriptions are not optional:

```python
# Weak — Claude guesses format (could return "Jan 1, 2025", "2025/01/01", etc.)
"start_date": {"type": "string"}

# Strong — description specifies format and edge case handling
"start_date": {
    "type": "string",
    "description": "Contract start date as YYYY-MM-DD. If only month/year given, use the 1st."
}
```

### Use `enum` for constrained values

```python
"priority": {
    "type": "string",
    "enum": ["critical", "high", "medium", "low"],  # without enum, Claude invents "urgent", "p0", etc.
    "description": "Priority level"
}
```

Enums prevent Claude from inventing values like "urgent", "p0", or "MEDIUM".

### Mark required vs. optional fields carefully

```python
{
    "type": "object",
    "properties": {
        "issue_id": {"type": "string"},   # always in the data
        "title":    {"type": "string"},   # always in the data
        "assignee": {"type": "string"},   # may not be set
        "due_date": {"type": "string"},   # may not be set
    },
    "required": ["issue_id", "title"]    # optional fields can be omitted by Claude
}
```

### Use nested objects for related fields

```python
"address": {
    "type": "object",
    "properties": {
        "street": {"type": "string"},
        "city": {"type": "string"},
        "state": {"type": "string"},
        "zip": {"type": "string"},
        "country": {"type": "string", "default": "US"}
    },
    "required": ["street", "city", "state", "zip"]
}
```

## Method 2: Direct JSON Prompting

When you cannot use the tools API (streaming contexts, some integrations), direct JSON prompting is the fallback:

```python
system = """You are a data extraction assistant.
IMPORTANT: Return valid JSON only. No text before or after. No markdown fences.

Schema:
{
  "sentiment": "positive" | "negative" | "neutral",
  "confidence": 0.0–1.0,
  "key_topics": ["string"],
  "requires_response": true | false
}"""

response = client.messages.create(
    model="claude-opus-4-6", max_tokens=512,
    system=system,
    messages=[{"role": "user", "content": customer_message}]
)

data = json.loads(response.content[0].text)  # may fail — wrap in try/except in production
```

<Callout variant="info" title="Add an assistant prefill to guarantee JSON starts">
You can add a partial assistant message to lock in the start of Claude's response:

```python
messages=[
    {"role": "user", "content": customer_message},
    {"role": "assistant", "content": "{"},  # Claude continues from here
]
```

This prevents Claude from starting with "Sure! Here's the JSON:" and guarantees the response starts with a JSON object.
</Callout>

- Streaming responses
- Simple schemas (1–2 levels of nesting, few fields)
- Contexts where the tools API isn't available

## Extracting Multiple Objects

To extract a list of structured items, wrap them in a top-level array schema:

```python
tools = [{
    "name": "extract_line_items",
    "description": "Extract all line items from the invoice. Return one object per item.",
    "input_schema": {
        "type": "object",
        "properties": {
            "line_items": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "description": {"type": "string"},
                        "quantity":    {"type": "number"},
                        "unit_price":  {"type": "number"},
                        "total":       {"type": "number"},  # Claude should compute qty × unit_price
                    },
                    "required": ["description", "quantity", "unit_price", "total"]
                }
            },
            "subtotal": {"type": "number"},
            "tax":      {"type": "number"},
            "total":    {"type": "number"},   # grand total including tax
        },
        "required": ["line_items", "total"]
    }
}]
```

<KeyTakeaway>
- `tool_use` is the most reliable mechanism for structured extraction — define a JSON Schema and set `tool_choice: {"type": "any"}`
- Tool input schemas support all JSON Schema primitives: `type`, `enum`, `required`, `description`, nested objects, arrays
- Use `description` fields extensively — Claude reads them to understand how to populate each field
- `enum` prevents Claude from inventing values for constrained fields
- Direct JSON prompting (system prompt + `json.loads()`) is the fallback for streaming or simple schemas
- Add an assistant prefill of `"{"` to prevent prose preamble before JSON
- For production pipelines, always use `tool_use` — prose instructions produce inconsistent output over time
</KeyTakeaway>
