The `Model Context Protocol (MCP)` is how Claude connects to external systems — databases, APIs, filesystems, anything. Instead of hard-coding integrations, you write an MCP server once and Claude can use it anywhere.

**The mental model:** Claude is a client. Your MCP server is a plugin. Claude calls your server's tools the same way it calls built-in tools — by name, with arguments, expecting a result.

---

## Build a Minimal MCP Server

This is a complete, working MCP server in Python. It exposes one tool: `get_weather`.

```python
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp import types

app = Server("weather-server")  # give your server a name

# Claude calls this to discover what tools exist
@app.list_tools()
async def list_tools() -> list[types.Tool]:
    return [
        types.Tool(
            name="get_weather",
            description="Get current weather for a city.",
            inputSchema={
                "type": "object",
                "properties": {
                    "city": {"type": "string", "description": "City name"},
                },
                "required": ["city"],  # Claude must supply this argument
            },
        )
    ]

# Claude calls this when it decides to use a tool
@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[types.TextContent]:
    if name == "get_weather":
        city = arguments["city"]
        # real implementation would call a weather API
        return [types.TextContent(type="text", text=f"72°F, sunny in {city}")]
    raise ValueError(f"Unknown tool: {name}")

if __name__ == "__main__":
    import asyncio
    asyncio.run(stdio_server(app))  # communicate via stdin/stdout
```

Install and run:

```bash
pip install mcp
python weather_server.py
```

## Connect It to Claude Code

Add the server to `~/.claude/claude_desktop_config.json` (or `.mcp.json` in your project):

```json
{
  "mcpServers": {
    "weather": {                              // name shown in Claude Code
      "command": "python",                   // how to start the server
      "args": ["/path/to/weather_server.py"] // script to run
    }
  }
}
```

That's it. Claude Code starts the process, discovers the tools via `tools/list`, and Claude can now call `get_weather` in any conversation.

<ExamTip>
The exam tests tools vs. resources: **tools** have side effects (writes, mutations, API calls). **Resources** are read-only data. Need to fetch a file for context? Resource. Need to write a file? Tool.
</ExamTip>

### Adding a Resource

Resources let Claude pull data into context without a tool call:

```python
# tells Claude what resources are available to read
@app.list_resources()
async def list_resources() -> list[types.Resource]:
    return [
        types.Resource(
            uri="db://schema/public",   # URI Claude uses to request this resource
            name="Database Schema",
            mimeType="application/json",
        )
    ]

# called when Claude wants the content of a specific resource
@app.read_resource()
async def read_resource(uri: str) -> str:
    if uri == "db://schema/public":
        return get_db_schema_json()     # return raw content as a string
    raise ValueError(f"Unknown resource: {uri}")
```

### Adding a Prompt Template

```python
# tells Claude what prompt templates exist
@app.list_prompts()
async def list_prompts() -> list[types.Prompt]:
    return [
        types.Prompt(
            name="code_review",
            description="Review code for bugs and style issues.",
            arguments=[
                types.PromptArgument(name="language", required=True),  # filled in at call time
            ],
        )
    ]

# called when Claude (or the user) invokes a prompt template
@app.get_prompt()
async def get_prompt(name: str, arguments: dict) -> types.GetPromptResult:
    lang = arguments["language"]  # e.g. "python", "typescript"
    return types.GetPromptResult(
        messages=[
            types.PromptMessage(
                role="user",
                content=types.TextContent(
                    type="text",
                    text=f"Review the following {lang} code for bugs and improvements:",
                ),
            )
        ]
    )
```

## Why All Three? A Code Review Assistant

Imagine you're building an MCP server for a GitHub code review assistant. Here's why you need all three primitives:

**The scenario:** A developer asks Claude — *"Review the open PR and leave comments on any bugs you find."*

To do that, Claude needs to:
1. Read the PR diff and file contents → **resource** (read-only, no side effects)
2. Post review comments back to GitHub → **tool** (mutates state, has side effects)
3. Apply a consistent review style every time → **prompt** (reusable template, not data or action)

```python
# ── RESOURCE: read the PR diff ──────────────────────────────────────────────
# Claude loads this into context before deciding what to do.
# It's read-only — fetching it doesn't change anything on GitHub.
@app.list_resources()
async def list_resources() -> list[types.Resource]:
    return [
        types.Resource(
            uri="github://pulls/42/diff",  # Claude requests this URI to get the diff
            name="PR #42 Diff",
            mimeType="text/plain",
        )
    ]

@app.read_resource()
async def read_resource(uri: str) -> str:
    pr_number = uri.split("/")[2]           # extract "42" from the URI
    return github_api.get_pr_diff(pr_number) # returns the raw diff text


# ── TOOL: post a review comment ──────────────────────────────────────────────
# Claude calls this to take action. It writes to GitHub — a real side effect.
# This can't be a resource because it mutates state.
@app.list_tools()
async def list_tools() -> list[types.Tool]:
    return [
        types.Tool(
            name="post_review_comment",
            description="Post an inline comment on a specific line of the PR.",
            inputSchema={
                "type": "object",
                "properties": {
                    "pr_number": {"type": "integer"},
                    "file":      {"type": "string", "description": "File path"},
                    "line":      {"type": "integer", "description": "Line number"},
                    "comment":   {"type": "string", "description": "Review comment text"},
                },
                "required": ["pr_number", "file", "line", "comment"],
            },
        )
    ]

@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[types.TextContent]:
    if name == "post_review_comment":
        github_api.post_comment(**arguments)  # actually writes to GitHub
        return [types.TextContent(type="text", text="Comment posted.")]


# ── PROMPT: consistent review style ─────────────────────────────────────────
# A reusable template that shapes HOW Claude reviews — not data, not an action.
# Using a prompt (vs. putting instructions in the system prompt) means any
# client can invoke this template by name, with their own arguments.
@app.list_prompts()
async def list_prompts() -> list[types.Prompt]:
    return [
        types.Prompt(
            name="pr_review",
            description="Review a PR diff for bugs, security issues, and style.",
            arguments=[
                types.PromptArgument(name="focus", required=False),  # e.g. "security" or "performance"
            ],
        )
    ]

@app.get_prompt()
async def get_prompt(name: str, arguments: dict) -> types.GetPromptResult:
    focus = arguments.get("focus", "general correctness")
    return types.GetPromptResult(
        messages=[
            types.PromptMessage(
                role="user",
                content=types.TextContent(
                    type="text",
                    text=(
                        f"Review this PR diff focusing on {focus}. "
                        "For each issue found, call post_review_comment with the file, "
                        "line number, and a clear explanation. Be specific and constructive."
                    ),
                ),
            )
        ]
    )
```

Notice how each primitive has a different job:

| | Resource | Tool | Prompt |
|---|---|---|---|
| **Reads data?** | Yes | Sometimes | No |
| **Changes state?** | Never | Yes | No |
| **Shapes behavior?** | No | No | Yes |
| **In this example** | Fetch PR diff | Post comment | How to review |

The rule of thumb: if Claude needs **information**, use a resource. If Claude needs to **do something**, use a tool. If you want Claude to **behave consistently**, use a prompt template.

## Transports: stdio vs. HTTP

| Transport | Use when |
|-----------|----------|
| **stdio** | Local process — Claude Code, CLI tools, scripts |
| **HTTP + SSE** | Remote server, cloud deployment, shared service |

The stdio server (shown above) is the simplest — Claude Code spawns your process and communicates over stdin/stdout. For a remote HTTP server:

```python
from mcp.server.sse import SseServerTransport
from starlette.applications import Starlette

transport = SseServerTransport("/messages")              # SSE endpoint path
starlette_app = Starlette(routes=transport.routes(app)) # wrap your MCP app in a web server
# deploy like any web app — uvicorn, gunicorn, etc.
```

## What Claude Actually Sees

Claude never sees JSON-RPC messages. The MCP client handles the protocol. Claude only sees the tool schema — name, description, and input shape — and decides when to call it based on context.

```
Your MCP server
    ↕  JSON-RPC over stdio / HTTP
MCP client  (built into Claude Code)
    ↕  tool schemas + results
Claude model
```

<ExamTip>
Claude sees tool schemas, not protocol details. The MCP client handles capability negotiation, JSON-RPC framing, and transport. Claude just calls tools by name.
</ExamTip>

<KeyTakeaway>
- An MCP server is a process that exposes tools, resources, and/or prompts over JSON-RPC
- Three primitives: `tools` (actions with side effects), `resources` (read-only context), `prompts` (templates)
- Connect to Claude Code via `claude_desktop_config.json` or `.mcp.json` — no client code needed
- Use **stdio** for local processes, **HTTP+SSE** for remote servers
- Claude sees only tool schemas — the MCP client handles all protocol details
- Design tools around agent tasks, not CRUD operations
</KeyTakeaway>
