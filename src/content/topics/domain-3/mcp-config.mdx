## `.mcp.json` — Project-Level Configuration

`.mcp.json` lives at the repository root and is committed to git. Every developer who clones the repo gets the same MCP server configuration:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    },
    "project-tools": {
      "command": "python",
      "args": ["-m", "tools.mcp_server"],
      "cwd": "${workspaceRoot}/tools"
    }
  }
}
```

The top-level key is always `"mcpServers"`. Each value is a server definition with:

## `~/.claude/settings.json` — User-Level Configuration

User-level MCP servers go in `~/.claude/settings.json` under the same `mcpServers` key:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me/projects"],
    },
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": {
        "BRAVE_API_KEY": "BSA..."
      }
    }
  },
  "theme": "dark",
  "model": "claude-opus-4-6"
}
```

These servers are available in every project. They combine with any `.mcp.json` found in the project — you get both.

## Environment Variable Substitution

Sensitive values (API keys, database URLs) should never be hardcoded in `.mcp.json` — it gets committed to git. Use environment variable references:

```json
{
  "mcpServers": {
    "stripe": {
      "command": "node",
      "args": ["./tools/stripe-mcp/index.js"],
      "env": {
        "STRIPE_SECRET_KEY": "${STRIPE_SECRET_KEY}",
        "STRIPE_WEBHOOK_SECRET": "${STRIPE_WEBHOOK_SECRET}"
      }
    }
  }
}
```

The `${VAR_NAME}` syntax substitutes from the shell environment at launch time. Developers set their own values in `.env` (gitignored) or in their shell profile:

```bash
# .env (gitignored)
STRIPE_SECRET_KEY=sk_test_...
STRIPE_WEBHOOK_SECRET=whsec_...

# Or in shell profile
export GITHUB_TOKEN=ghp_...
```

<Callout variant="info" title="Never commit secrets to .mcp.json">
`.mcp.json` is committed to git. Any literal API key in this file is exposed to everyone with repo access, including potentially public GitHub repositories. Always use `${ENV_VAR}` references for sensitive values.
</Callout>

## Transport Configuration

### stdio (default for local servers)

When `command` is specified, Claude Code spawns the process and communicates via stdin/stdout. This is the stdio transport — standard for local tools:

```json
{
  "mcpServers": {
    "my-tool": {
      "command": "python",
      "args": ["./mcp_server.py"]
    }
  }
}
```

### HTTP + SSE (for remote servers)

For servers running as HTTP services (remote or in a container), use the `url` field instead of `command`:

```json
{
  "mcpServers": {
    "remote-analytics": {
      "url": "https://mcp.internal.company.com/analytics",
      "headers": {
        "Authorization": "Bearer ${ANALYTICS_TOKEN}"
      }
    }
  }
}
```

## Complete `.mcp.json` Example

A realistic project configuration with multiple servers:

```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres-dev": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-postgres",
        "${DATABASE_URL}"
      ]
    },
    "project-scripts": {
      "command": "node",
      "args": ["./scripts/mcp-tools/index.js"],
      "cwd": "${workspaceRoot}",
      "env": {
        "NODE_ENV": "development",
        "LOG_LEVEL": "info"
      }
    },
    "docs": {
      "url": "https://docs-mcp.internal.company.com",
      "headers": {
        "X-API-Key": "${DOCS_MCP_KEY}"
      }
    }
  }
}
```

## Server Naming and Tool Namespacing

When multiple MCP servers are configured, their tools are namespaced by server name to avoid collisions. If two servers both expose a `search` tool:

```
github/search       ← from the "github" server
docs/search         ← from the "docs" server
```

This is why server names in the config key matter — use descriptive, stable names. Renaming a server name in `.mcp.json` changes how its tools are referenced.

## Verifying MCP Configuration

```bash
# List all configured MCP servers and their status
claude mcp list

# Add a server interactively
claude mcp add

# Test a specific server
claude mcp test github

# View server logs for debugging
claude mcp logs github
```

<KeyTakeaway>
- `.mcp.json` in repo root = project-level config, committed to git, shared with team
- `~/.claude/settings.json` mcpServers = user-level config, personal, available in all projects
- stdio transport: use `command` + `args` to spawn a local process
- HTTP/SSE transport: use `url` + optional `headers` for remote servers
- Use `${ENV_VAR}` syntax for secrets — never hardcode API keys in `.mcp.json`
- Server names in config become the namespace for that server's tools
- `claude mcp list` / `claude mcp test <name>` for verifying and debugging configuration
</KeyTakeaway>
