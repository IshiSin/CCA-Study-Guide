`Plan Mode` is a Claude Code feature that separates the *thinking* phase from the *acting* phase. In Plan Mode, Claude reasons about what it would do and explains the plan — but takes no actions. No files are modified, no commands are run, no tools are called with side effects.

## Why Plan Mode Exists

Claude Code's default behavior is to execute immediately: read files, make edits, run commands. This is efficient for well-scoped tasks but creates risk for:
- Large refactors touching many files
- Destructive operations (deletes, database changes)
- Tasks where the approach matters as much as the outcome
- Situations where you want to review strategy before investing in execution

Plan Mode lets you validate Claude's approach before it acts.

---

## Entering Plan Mode

`Keyboard shortcut:` `Shift + Tab` toggles Plan Mode on and off.

When Plan Mode is active, Claude Code shows a visual indicator, and Claude's responses describe what it *would* do rather than doing it:

```
[Plan Mode ON]

User: Refactor the authentication middleware to support OAuth2

Claude: Here's how I'd approach this refactor:

1. **Read current auth middleware** — src/middleware/auth.ts
   Understand the current JWT validation logic and what it exports.

2. **Read existing route usages** — grep for `authMiddleware` across the codebase
   Identify all routes currently protected, to ensure none break.

3. **Create OAuth2 strategy** — src/middleware/oauth2.ts (new file)
   Implement PKCE flow using the `passport-oauth2` library.

4. **Update auth middleware** — src/middleware/auth.ts
   Add strategy selection based on request headers or route config.
   Keep JWT path working — backward compatibility.

5. **Update 3 affected routes** — src/routes/api.ts, admin.ts, webhooks.ts
   Add `strategy: 'oauth2'` option where needed.

6. **Add tests** — src/middleware/__tests__/auth.test.ts
   Test both JWT and OAuth2 paths. Mock the OAuth provider.

Does this approach look right? Any constraints I should know about
(e.g., which routes need OAuth2 vs. JWT)?
```

<ExamTip>
The exam tests Plan Mode as a tool for human review before execution — not as a way to make Claude "think harder." The key distinction: Plan Mode prevents side effects. Claude can still read files (reads are not side effects) but will not write, edit, or run commands that modify state.
</ExamTip>

## Plan Mode Workflow

The typical workflow is: plan → review → approve → execute.

```
Step 1: Enter Plan Mode (Shift+Tab)
Step 2: Describe the task
         → Claude reads the codebase, describes the approach
Step 3: Review the plan — ask questions, request changes
         → "Don't touch the webhooks route"
         → "Use the existing OAuth library we already have"
Step 4: Exit Plan Mode (Shift+Tab again)
Step 5: Claude executes — following the refined plan
```

This is particularly valuable for:

```markdown
## When to use Plan Mode

Large refactors:
  "Extract the payment logic from OrderService into a dedicated PaymentService"
  → See which files will be touched before any are modified

Database changes:
  "Add soft deletes to the User model"
  → Review migration strategy and backward compatibility before running migrations

Uncertain scope:
  "Clean up the auth code"
  → Discover what Claude considers "auth code" and what it plans to do
     before it starts making changes you might not want

High-stakes tasks:
  "Delete all the deprecated API endpoints"
  → Confirm exactly which endpoints Claude considers deprecated
```

## Plan Mode in Automated Contexts

In the `-p` (print/non-interactive) mode, Plan Mode can be engaged explicitly. When building pipelines that use Claude Code, you can use Plan Mode to generate a plan as structured output, have a human or another system approve it, then execute:

```bash
# Phase 1: Generate plan only (no execution)
claude --plan-only -p "Migrate the database schema to add soft deletes" \
  --output-format json > migration_plan.json

# Human reviews migration_plan.json

# Phase 2: Execute the approved plan
claude -p "Execute the migration plan in migration_plan.json"
```

<Callout variant="info" title="Plan Mode is a checkpoint, not a guarantee">
A plan describes Claude's intended approach at the moment of planning. When execution begins, Claude may encounter new information (a file has different contents than expected, a test fails) and adapt. The plan is a starting point for alignment, not a rigid contract.
</Callout>

## Asking for Plans Without Plan Mode

You can also get plan-style reasoning without toggling Plan Mode by asking directly:

```
"Before making any changes, describe what you would do to refactor
the authentication middleware. Don't make any edits yet."
```

This is less reliable than Plan Mode (Claude might still make an edit) but useful when you want a quick explanation before a small task.

<KeyTakeaway>
- Plan Mode separates reasoning from acting — Claude reads and thinks but does not write or run commands
- Toggle with `Shift + Tab` in the Claude Code UI
- Use before large refactors, destructive operations, or any task where approach matters
- Claude can read files in Plan Mode (read-only tools are allowed); file edits and shell commands are blocked
- Workflow: plan → review → refine → exit Plan Mode → execute
- The plan is an alignment checkpoint, not a rigid contract — Claude adapts during execution
</KeyTakeaway>
